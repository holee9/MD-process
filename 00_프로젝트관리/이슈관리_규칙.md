# 이슈 관리 및 스케줄링 운용 규칙

## 목적
스케줄링을 통한 업무규칙 개발 과정에서, 작성·보강되는 문서 이력을 GitHub Issue로 관리하여
누락·지연 작업을 추적·보완한다.

## 관리 흐름

1. **문서 작업** — Claude 스케줄링 세션이 해당 카테고리 폴더(예: `03_설계_개발관리/`)에 문서 작성/수정
2. **이슈 드래프트 작성** — 작업 항목을 `issue-drafts/NNN_*.md`에 frontmatter 포함 작성
3. **커밋 & push** — Claude가 `main`으로 push
4. **자동 이슈 생성** — `.github/workflows/auto-issue.yml`이 트리거되어 이슈 생성/업데이트
5. **이력 기록** — `issue-drafts/_log.json`에 파일명 ↔ 이슈번호 매핑 저장
6. **후속 점검** — 다음 스케줄 세션에서 열린 이슈 목록을 기반으로 누락 작업 보완

## 이슈 분류 라벨

| 라벨 | 용도 |
|------|------|
| `documentation` | 규정/절차 문서 작성 |
| `review` | 교차검증·점검 결과 |
| `research` | 리서치 결과 반영 필요 |
| `maintenance` | repo·도구·환경 유지보수 |
| `compliance` | 법규/규제 대응 |
| `qms` | QMS 프로세스 관련 |
| `design-control` | 설계·개발 관리 |
| `risk` | 위험관리 (ISO 14971) |

## 네이밍 규칙

`NNN_카테고리_간단제목.md`
- `NNN` — 3자리 일련번호 (001부터)
- `카테고리` — `git`, `qms`, `mfds`, `mdr`, `fda`, `iso13485`, `iso14971`, `pms` 등
- `간단제목` — 한글 허용, 공백 대신 `_`

예: `001_git_repo_연동점검.md`, `002_qms_문서관리_절차초안.md`

## 상태 전이

```
  draft(frontmatter state 미지정) → open(자동) → closed(수동 state: closed)
                                        ↑             |
                                        └── state: open ┘
```

## 운영 주의

- 파일명은 한 번 정하면 변경 금지(이슈 매핑 키).
- 이슈 번호는 `_log.json`에서 확인.
- 워크플로 실행 실패 시 `Actions` 탭에서 로그 확인 후 재push로 재시도.
